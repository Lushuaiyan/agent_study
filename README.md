### 1-纯api调用
这部分的主要作用是:
1. 了解如何连接LLM
2. 了解API的输入格式和响应格式
3. 为后续学习agent框架打好基础

#### 1.创建客户端
连接LLM的第一步是创建客户端
```python
from openai import OpenAI
client = OpenAI(
    api_key="your_key",
    base_url="模型服务地址"
)
```
创建客户端有以下好处
1. **集中管理连接配置**
    - 可以把url, api_key等连接相关配置一次性设置好, 后续调用不需要重复配置
    > 在创建客户端时, 有几个参数是很常用的
    > 1. api_key: 必填, 相当于身份认证, 没有它api会拒绝你的调用
    > 2. base_url: 指定API的基础服务地址, 默认是openai的服务地址, 使用哪个厂商, 就填谁的服务地址
    > 3. timeout: 请求超时时间, 是计算发出请求到接到响应的时间总和(可以使用httpx库单独配置握手超时时间, 响应超时时间), 因此如果需要生成非常长的响应, 最好使用流式输出
    > 4. max_retries: 最大重试请求次数, 总请求次数=原始次数+最大重试次数, 并非所有报错都会重试, 如4xx客户端错误和某些不可重试的网络异常都不会重试
2. **复用底层http连接, 提升性能**
    - client内步维护了连接池, 多次请求会使用同一个TCP连接, 减少握手开销
3. **提供统一的接口和更好的代码组织**
    - 所有的API功能都挂载在client对象下, 链式调用, 代码结构更清晰
4. **支持多实例, 多配置并行**
    - 可以创建多个客户端, 分别连接不同服务, 互不干扰

#### 2.OpenAI客户端的常用接口
1. `client.chat.completions.create()`：最核心、最常用的接口。用于发起对话补全请求，适合多轮对话等场景。
    - `.chat`是一个命名空间, 表明这是一个对话的交互, `.completions`则会进一步确认任务类型是文本补全, `.create`是最终发送请求
    - `.create`有许多参数, 其中最重要的是:
        1. `model`: 用来指定要使用的模型名称
        2. `messages`: 对话历史信息记录, LLM生成响应的依据, 它是一个消息对象列表, 每个对象包含 `role` 和 `content`
        3. `stream`: 是否使用流式输出, 返回的结果按token返回
        4. `tempurature`: 控制生成文本的随机性和创造性, 取值范围通常是0到2, 值越低, 输出越确定和保守; 值越高, 输出越多样和出人意料
2. `client.responses.create()`：OpenAI 推荐的新版主要API，功能更强大。它统一了文本生成接口，通过 input 参数传入提示词。
3. `client.embeddings.create()`：用于将文本转换为向量（Embeddings）。这是构建RAG（检索增强生成）应用的基础。
4. `client.images.generate()`：用于根据文本描述生成图像。
5. `client.fine_tuning.jobs.create()`：用于创建和管理模型微调任务，让模型在特定任务上表现更好。
    - 微调是在创建agent之前需要处理的任务, 相当于对模型进行二次训练, 需要准备质量好数量多的json格式的训练数据, 能够提高对指定问题的响应准确率
6. `client.moderations.create()`：用于内容审核，检查文本是否违反OpenAI的使用政策。
7. `client.files.create()`：用于上传文件，例如为微调任务准备数据。
> 我使用DeepSeek作为学习的模型, 其中的一些接口不适配, 如: responses接口, images接口, moderations接口和files接口

##### 响应解析
流式响应和非流式响应的结果不同
非流式响应的结果是一个完整的对象, 直接从其中提取结果即可
流式响应的结果是一个可迭代的生成器对象, 需要迭代读取结果即可
响应特点: 生成对象基础信息(id, model, created, usage和choices等), 其中最重要的是choices, 是响应的具体内容
choices是一个列表, 通常只有一个元素, `Choice`对象
流式响应和非流式响应的差异主要在`Choice`中
- **相同点**: `index`是索引, 一般是0, `finish_reason`是结束标记, 非流式响应默认为"stop", 流式响应默认为None, 最后一个响应为"stop", `message`或`delta`是返回信息的对象, 流式是`delta`, 非流式是`message`
- **非流式响应**: `message`中的`content`是LLM的最后结果, `thinking_content`是LLM思考的结果, `role`是响应的角色, 一般是"assistant"
- **流式响应**: `delta`中的数据块严格按照时间顺序排列, 先生成`thinking_content`, 再生成`content`, 绝对不会交叉, 只有第一个数据块有`role`, 通常也是"assistant"

因此, 获取LLM响应输出结果的方式是:
- **非流式**: `response.choices[0].message.content`
- **流式**: `for chunk in response: content += chunk.choices[0].delta.content`

### 2-记忆
所有的模型都是**无状态的**, 所以每一次和模型进行交互, 都是一次单独的对话, 没有之前的记忆

但是记忆是模型解决复杂问题的重要工具, 因为复杂任务通常不是一次会话就可以实现的, 需要一步一步的进行, 稳步推进目标, 最终实现任务, 因此记忆系统至关重要

一般管理会话记忆有以下方式:
1. **全量传输**: 保存所有会话信息, 将其保存成一个文件, 或保存到数据库
2. **分层记忆架构**: 模仿人类的记忆结构, 将记忆分成不同层级管理, 如短期记忆, 长期记忆, 工作记忆, 情景记忆等
3. **语义检索和向量数据库**: 将记忆片段通过embedding模型转化成向量, 当用户发起新查询时, 系统将其向量化, 并在向量数据库中检索语义最相似的“记忆”, 检索到的相关内容会被注入到当前Prompt中, 作为上下文
4. **记忆的总结压缩和折叠**: 对冗长的记忆历史进行提炼总结

> 其实全量传输就是解决记忆的基础且最重要的方法, 其它所有的方法都会导致信息丢失, 本质上都是在尽可能不丢失信息的情况下, 维持记忆的妥协

### 3-上下文窗口管理
上下文窗口管理是agent记忆功能的延申, 为了更高效的利用有限的上下文窗口, 有许多方法:
1. **滑动窗口**: 当上下文窗口不够时, 会放弃最早的记忆(也有放弃中间的记忆,保留最早记忆的方法, 为了锚定任务目标)
2. **会话压缩**: 当即将达到上下文窗口的上限时, 对整个会话信息进行压缩, 只保留关键信息
3. **按需提取**: 将所有记忆信息向量化, 当用户需要时, 通过向量召回, 问题是召回率很难提升, 记忆本身很少使用这种方法, 一般是知识库使用这种方法充当上下文
    > 关于按需提取事实上不止有rag这种方法, 也可以直接使用精确检索的方式, rag主要的工作范围是语义检索
4. **会话隔离**: 将会产生大量上下文的任务分配给子agent完成, 主agent只用来分配任务
5. **分层记忆**: 按照不同的层级管理记忆, 使用结构化的方式完成记忆的精简

> 真实的项目中常常是这些方法的混合

### 4-工具
**LLM**只是一个对话生成器, 根据之前训练的数据回答用户的问题,本质上是一个自产自销的过程, 没有影响外部世界的手段
工具就是LLM的手脚, 能够帮助LLM做到之前不能做到的事情

**tools**在openai中是**Function Calling**(因为openai中的工具都是函数)
Function Calling并非让LLM直接执行代码, 而是一种让LLM以结构化, 机器可读的方式请求调用某个函数的能力
当LLM决定使用工具时, 响应结果中的message.content为None, 重点在message.tool_calls字段, 是LLM请求调用的工具

- **工作原理**: 在调用API时将工具的名称, 描述和参数结构(JSON Schema)作为tools参数传给模型
- **模型决策**: LLM根据用户的提问, 自主决定是否需要调用某个工具, 并返回一个标准化的tool_calls对象, 包含需要调用的函数名和所需的参数(json格式)
- **你的职责**: 你的代码解析该响应, 然后真正地执行对应函数, 并将结果返回给LLM, 让他生成最终的自然语言回复

典型的工具定义流程
```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "函数名",
            "description": "该工具的描述, 包含具体作用, 使用场景等",
            "parameters": {
                "type": "object",
                "properties": {
                    "参数名1": {
                        "type": "参数数据类型(str, int等)",
                        "description": "参数的描述"
                    },
                    "参数名2":{
                        "type": "参数数据类型(str, int等)",
                        "enum": ["可选值1", "可选值2"],
                        "description": "参数的描述"
                    }
                },
                "required": ["必需参数1", "必需参数2"]
            }
        }
    }
]
```

以上就是一个工具函数的定义过程, 这些工具的定义本身会消耗宝贵的上下文窗口, 因此不能无限制的注册工具, 一般推荐注册10个左右(超过10个选择工具也会出现LLM选择困难的问题)
通常如果工具过多, 应该对工具进行逻辑分组, 通过路由让模型先选组再选具体工具

> 注意: openai库是没有能力调用你的函数的, 它只负责收发文件, 因此调用函数的任务也要通过代码实现

### 5-规划
**规划**(Planning)是agent从"被动响应"到"主动解决复杂问题"的飞跃, 让agent能像人类一样在行动前先拆解目标, 制定步骤

- **定义**: 将用户的复杂,模糊的指令, 分解成一系列可执行的, 有序的子任务
- **与工作流的区别**: 规划是动态的, 可以根据中间的结果调整, 工作流是静态预定
- **核心价值**: 避免长上下文中的迷失, 提升复杂任务的成功率

#### 实现规划的模式

1. **ReAct**: 推理+行动, 隐式规划, 边做边想, (Thought->Action->Observation循环), 任务步骤不确定, 需要探索性执行
2. **Plan-and-Solve**: 先规划后执行, 显示规划, 首轮生成完整步骤清单, 再按序执行, 任务结构清晰, 子任务相对独立

> 这里主要实现Plan-and-Solve+反思(Reflection)的模式

#### 实现流程
实现目标: 实验一个Planner类, 核心职责是生成计划和更新计划状态
1. 定义计划数据结构(Pydantic模型): 让LLM输出结构化JSON, 保证程序可靠解析
2. 编写规划提示词: 向LLM解释需要输出的格式,并明确要求只输出JSON
3. 实现执行循环: 执行循环, 一步一步的解决每个步骤, 注意失败重规划, 继续生成剩余计划(动态调整), 最后汇总所有步骤结果, 调用LLM生成最终回复
4. 关键技巧-上下文隔离: 在Plan-and-Solve中, 不要把整个计划的所有历史信息都塞给执行步骤的LLM, 仅传入原始目标+当前步骤描述+该步骤依赖的前置步骤摘要
